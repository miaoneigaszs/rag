# 工程化改造说明

## 目标

本次改造的目标不是单纯“让功能能跑”，而是把项目收敛到可长期维护、可持续演进、可逐步上线的工程基线。核心关注点有四类：

1. 依赖边界清晰，轻量模块与测试不被重型依赖绑死。
2. 文档身份稳定，索引、删除、重建索引行为一致可预期。
3. 检索状态可回滚，长期运行不会因删除/重建而持续漂移。
4. 源码与测试全部回到 UTF-8 正常状态，避免后续协作继续污染文本。

## 本次已完成的关键改造

### 1. 文档身份与重建索引

- `DocumentChunk` payload 统一同时保存 `source_file` 与 `source_path`。
- 删除文件时优先按 `source_path` 删除，只有兼容旧数据时才回退到 `source_file`。
- `RAGEngine.index_file()` 在同一路径重建索引前，会先按 `source_path` 清理旧数据，再写入新 chunk。

这样可以避免以下问题：

- 不同目录下同名文件互相串档。
- 文件内容变更后重新入库，旧 chunk 残留。
- 依赖 basename 删除导致误删其他文档。

### 2. `extra_meta` 安全边界

- `DocumentChunk.extra_meta` 现在禁止覆盖核心保留字段。
- 如果调用方试图覆盖 `doc_id`、`source_path`、`section_index` 等关键字段，会直接抛出异常。

这样可以保证过滤、删除、溯源、section 扩展逻辑不会被外部元数据悄悄破坏。

### 3. BM25 生命周期一致性

- `SparseEncoder` 已支持 `update_idf()` 与 `remove_idf()` 的对称更新。
- `QdrantVectorStore` 在按条件删除点位时，会先回滚对应文本对 IDF 的贡献。

这保证了长期运行中，删除文档不会让稀疏检索统计量持续漂移。

### 4. 依赖懒加载与可选化

以下依赖已从“模块导入时强依赖”改为“对象实例化时按需依赖”：

- `openai`
- `httpx`
- `qdrant-client`

这带来的直接收益：

- 只跑配置、模型、切块、BM25 等轻量测试时，不需要完整安装在线依赖。
- `import rag` / `import rag.engine` 的失败面更小。
- 依赖问题更容易定位到具体组件，而不是在包导入阶段整体炸掉。

### 5. Contextual Retrieval 缓存键增强

- cache key 现在基于完整 `section_text`、`chunk_content`、`model`、`prompt_version` 计算。
- 不再只截取前 200/500 字符，避免正文后半段变化却命中旧缓存。

### 6. UTF-8 与中文乱码修复

已修复并重写下列核心源码/测试中的乱码与破损字符串：

- `rag/config.py`
- `rag/contextual.py`
- `rag/embedder.py`
- `rag/engine.py`
- `rag/models.py`
- `rag/reranker.py`
- `rag/vector_store.py`
- `tests/` 下相关测试与示例文本

目前仓库主干源码与测试中的明显 `?` 式乱码已经清理到可维护状态。

## 当前工程基线

### 测试状态

当前测试结果：

- `78 passed`
- `1 skipped`

跳过项是 `docling` 相关图片提取测试。该测试被设计为依赖可选组件，未安装 `docling` 时跳过属于预期行为。

### 依赖策略

推荐将依赖拆分为三层：

1. 核心本地开发依赖：`pytest`、`tenacity`、`tqdm` 等。
2. 在线能力依赖：`openai`、`httpx`。
3. 基础设施依赖：`qdrant-client`、`redis`、`diskcache`、`docling`、`unstructured`。

这样做可以让 CI 以更低成本覆盖核心逻辑，同时把外部基础设施测试放进单独 job。

## 后续建议

### 1. 继续拆分测试矩阵

建议至少拆成三组：

- `unit`: 纯本地、无外部依赖。
- `integration`: Qdrant / Redis / Unstructured。
- `optional`: Docling、多模态、真实 API 冒烟。

### 2. 补上线前观测能力

建议增加：

- 统一结构化日志字段：`doc_id`、`source_path`、`query_id`。
- 指标：索引耗时、embedding 失败率、rerank 失败率、缓存命中率。
- 健康检查：Qdrant 连通性、模型 API 可达性、缓存后端状态。

### 3. 固化依赖与环境

建议补齐：

- `requirements-dev.txt` 或 extras 分组。
- CI 中的 Python 版本矩阵。
- 本地开发 `.env.example`。
- 部署环境的配置清单与最小权限原则。

### 4. 增加回归测试

后续优先补这些回归点：

- 同一路径重复索引后的旧数据清理。
- 删除文档后 BM25 状态回滚。
- `delete_file()` 对绝对路径、相对路径、legacy basename 的行为差异。
- Contextual cache 在 section 尾部变化时的失效。

## 结论

经过这次改造，项目已经从“功能型原型”明显前进到“可维护工程项目”的层级：

- 核心检索链路仍保持原有能力。
- 依赖边界、索引身份、缓存与删除一致性显著增强。
- 中文源码和测试文本恢复为正常 UTF-8，可继续协作。

如果下一步继续补齐 CI、依赖分层、观测与集成测试，这个仓库就具备比较像样的工业化演进基础了。
